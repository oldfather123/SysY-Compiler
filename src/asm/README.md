# ASM Backend README

这个目录放的是当前的 RISC-V 后端实现。目前目标很直接：**从现有 IR 直接翻译到 RISC-V 汇编，先把结果跑对，不追求生成代码性能。**

这套实现是比赛里常见的“最短可用路径”：

- 不引入 Machine IR。
- 不做寄存器分配。
- 不做指令调度。
- 不做 peephole 优化。
- 大多数 IR 值都先落栈，再从栈里取出来参与下一条指令。

也就是说，当前版本的重点不是“汇编好看”，而是“语义别错、调用约定别错、测试能过”。

## 1. 目录内容

- `AsmEmitter.h` / `AsmEmitter.cpp`
  直接把 `CodeGen` 产出的 IR 翻译成 RISC-V 汇编文本。
- `sylib.s`
  SysY 运行时库的 RISC-V 汇编实现。

## 2. 整体流程

当前后端的真实流水线是：

```text
SysY 源程序
  -> AST
  -> 语义检查
  -> CodeGen 生成 IR
  -> AsmEmitter 直接把 IR 打印成 .s
  -> RISC-V GCC 只负责把 .s + sylib.s 汇编/链接成 ELF
  -> QEMU 运行 ELF
  -> 收集程序输出和返回码
  -> 和标准答案比较
```

仓库里实际生成的文件扩展名是小写 `.s`。如果下面文字里偶尔口语化地说“.S”，都指的是这份 RISC-V 汇编输出文件。

这里要特别区分两件事：

- **`.sy -> .s` 是我们自己的编译器做的**，核心逻辑就在 `CodeGen` 和 `AsmEmitter`。
- `asm_test.sh` 里的 `riscv64-linux-gnu-gcc` **不是在帮我们生成目标汇编**，它只是把已经生成好的 `.s` 汇编并链接起来，这一步不属于“拿主机编译器代写后端”。

## 3. 这个“直接翻译，能跑就行”的版本怎么实现

## 3.1 不做寄存器分配，统一走栈

当前后端最核心的取舍就是：**几乎每个非 `void` 的 IR Op 都分配一个栈槽。**

这样做的好处是：

- 实现简单。
- 不需要活跃变量分析。
- 不需要图着色或线性扫描寄存器分配。
- 调试时容易定位每个 IR 值最后放到了哪里。

代价就是：

- load/store 非常多。
- 汇编性能一般。
- 栈帧可能很大。

但对“先过功能测试”来说，这个代价是可以接受的。

## 3.2 AsmEmitter 的工作方式

`AsmEmitter` 是一个**单遍直接打印汇编**的后端。

它做的事情大致分成两段：

### 第一段：收集模块信息

- 扫描整个 IR `Region`。
- 找出全局变量。
- 找出每个函数对应的基本块范围。
- 如果全局初始化里除了 `GlobalOp` 之外还有别的 IR，就额外生成一个 `__sysy_global_init` 函数。
- 生成 `main` 时，如果模块里存在这类全局初始化逻辑，会先 `call __sysy_global_init` 再执行用户的 `main` 函数体。

### 第二段：逐函数输出汇编

每个函数在真正输出前先做一次 `preScanFunction`：

- 统计局部 `alloca` 需要多少空间。
- 统计这一个函数里最大的栈上传参数量。
- 给每个有返回值的 IR Op 分配 spill slot。
- 计算总栈帧大小，并按 16 字节对齐。

然后再真的输出：

- 函数标签。
- 序言。
- 每个基本块的标签。
- 每个 IR Op 对应的 RISC-V 指令。
- 返回时输出尾声。

## 3.3 当前栈帧布局

当前实现里，一个函数的栈帧从低地址到高地址大致是：

```text
sp + 0
  outgoing stack args area
  保存给被调用函数的栈上传参

sp + outgoingSize
  ra

sp + outgoingSize + 8
  s0

再往上
  alloca 区

再往上
  每个 IR 结果值的 spill slot

sp + frameSize
  旧的 sp
```

这个布局的特点是：

- **先给调用别人预留传参区**，避免一次函数里多个调用互相覆盖参数位置。
- `alloca` 和临时值都在本函数栈帧里。
- 对超过 12 位立即数范围的栈偏移，`AsmEmitter` 会先用 `li/add` 算地址，再做 `ld/lw/sd/sw/flw/fsw`，所以超大栈帧也能处理。

## 3.4 直接翻译规则

当前汇编生成规则很朴素：

- `IntOp`：`li` 到整数寄存器，再写回自己的栈槽。
- `FloatOp`：先把浮点常量转成 IEEE 754 位模式，再 `fmv.w.x` 放进浮点寄存器。
- 二元整数运算：从两个栈槽 load 到 `t0/t1`，算完再存回结果栈槽。
- 二元浮点运算：从两个栈槽 load 到 `ft0/ft1`，算完再存回结果栈槽。
- `LoadOp` / `StoreOp`：先拿到地址，再按标量类型做 `lw/sw`、`ld/sd` 或 `flw/fsw`。
- `CallOp`：整数参数放 `a0-a7`，浮点参数放 `fa0-fa7`，放不下的部分写到当前函数预留的 outgoing arg area。
- `ReturnOp`：整数返回走 `a0`，浮点返回走 `fa0`。
- `BranchOp`：条件值先变成整型布尔，再用 `bnez` 跳转。

这套做法很“笨”，但它和当前 IR 的结构贴得很紧，所以改 bug 时比较直接。

## 4. asm_test.sh 是怎么工作的

脚本位置在 [`src/scripts/asm_test.sh`](../scripts/asm_test.sh)。

它支持两种入口：

```bash
src/scripts/asm_test.sh test/2026/functional
src/scripts/asm_test.sh test/2026/functional/00_main.sy
```

也就是：

- 可以测一个目录下的全部 `.sy`
- 也可以只测单个 `.sy`

## 4.1 它的执行步骤

对每个用例，脚本会做这几步：

### 第 1 步：调用我们的编译器生成 `.s`

```bash
src/build/compiler input.sy /dev/null --dump-asm output.s
```

这里真正完成了：

- `SysY -> AST`
- `AST -> IR`
- `IR -> RISC-V .s`

其中最后一步是这个目录里的后端实现完成的。

### 第 2 步：把 `.s` 和 `sylib.s` 汇编并链接

脚本会调用：

- `RV_GCC`，默认是 `riscv64-linux-gnu-gcc`

把两份汇编：

- 编译器生成的 `case.s`
- 运行时库 `src/asm/sylib.s`

链接成一个 RISC-V 可执行文件。

这一步只是：

- 汇编
- 链接

**不是**在替我们生成目标代码。

### 第 3 步：用 QEMU 跑这个 RISC-V 可执行文件

脚本默认使用：

- `qemu-riscv64-static`

来运行上一步产出的 ELF。

为了避免大栈帧程序因为 QEMU 默认访客栈太小而崩掉，脚本还会传：

- `-s "$QEMU_STACK_SIZE_BYTES"`

默认值是 `67108864`，也就是 64 MiB。

### 第 4 步：收集“运行结果”

脚本收集到的结果不是只有标准输出，还包括**退出码**。

最终写入 `src/test_output/.../asm/out/<case>.out` 的内容格式是：

```text
程序标准输出
程序退出码
```

如果程序标准输出最后没有换行，脚本会先补一个换行，再把退出码附到最后一行。

这样做是为了和项目里解释执行器的输出格式保持一致。

### 第 5 步：和标准答案比较

脚本最后会把：

- 期望输出：`test/.../<case>.out`
- 实际输出：`src/test_output/.../asm/out/<case>.out`

做一次 `diff -u` 比较。

比较前会先用 `awk` 去掉行尾的 `\r`，避免 Windows 风格换行造成假失败。

## 4.2 asm_test.sh 的输出文件

脚本会在：

```text
src/test_output/<target-dir>/asm/
```

下面写出这些文件：

- `asm/<case>.s`
  你的编译器生成的 RISC-V 汇编。
- `exe/<case>`
  汇编并链接后的 RISC-V 可执行文件。
- `out/<case>.out`
  QEMU 跑出来的实际输出，最后一行附带退出码。
- `err/<case>.err`
  编译、链接、运行阶段的标准错误。
- `diff/<case>.diff`
  和标准输出不一致时留下的 diff。

所以如果一个用例没过，最常看的就是：

- 先看 `err/<case>.err`
- 再看 `asm/<case>.s`
- 最后看 `diff/<case>.diff`

## 5. CodeGen 做了哪些关键修复

为了让这个直接翻译版真正跑通 `functional` 和 `h_functional`，`CodeGen` 侧补了几类关键问题。

## 5.1 先发全局，再发函数

一开始如果把全局初始化和函数体穿插着往 IR 里塞，`AsmEmitter` 很容易把函数块误判进全局初始化区域，或者让多个函数的边界不清楚。

修复后变成：

- 先遍历所有顶层 `VarDecl`
- 再遍历所有顶层 `FunctionDecl`

这样模块结构稳定很多，汇编侧也更容易正确分段。

## 5.2 条件表达式统一先布尔化

之前 `if` / `while` / `!expr` 如果直接拿原始值做判断，会在下面这些情况下出错：

- `float` 条件
- 指针条件
- 需要先比较再跳转的表达式

现在 `CodeGen` 会先用 `emitBoolExpr` / `toBoolValue` 把条件统一规整成 `0/1`：

- `int` / 指针：非零即真
- `float`：和 `0.0` 比较，非零即真

这样控制流分支就统一了。

## 5.3 `&&` 和 `||` 改成真正的短路求值

一开始如果把逻辑与/或直接当普通二元运算处理，就会出现：

- 右侧本来不该执行却被执行了
- 带副作用表达式结果错误

现在的做法是：

- 给逻辑表达式临时分配一个结果地址
- 建立 `rhs`、`short`、`end` 三类基本块
- 左值先判断
- 需要短路时直接写入 `0` 或 `1`
- 不短路时再去算右值

这才符合 C/SysY 语义。

## 5.4 修了数组和指针的类型降低

在直接后端里，数组和指针本质上都应该作为“地址值”参与后续翻译。

因此 `CodeGen::toValueType` 现在会把：

- 指针
- 数组

都映射成整数类值，让后端把它们当地址来传递、存储和做地址运算。

这对下面几类场景都很重要：

- 数组参数传递
- 子数组传参
- 多维数组寻址
- 浮点数组参数避免被错误当成浮点标量

## 5.5 修了多维数组下标地址计算

多维数组最容易错的地方是 stride。

当前修复点是把：

- 真数组
- 指针退化后的数组参数

这两种情况区分开来计算 stride，避免 `arr[i][j]`、`a[i]` 这类表达式把偏移算错。

## 5.6 `arr[i]` 如果还是数组，就返回地址而不是立刻 load

在多维数组里，`arr[i]` 可能表示的是“第 i 行子数组”，不是一个标量元素。

如果这时直接 `LoadOp`，就会把“子数组地址”误当成“标量值”，后续传参和寻址都会错。

修复后：

- 如果 `ArraySubscriptExpr` 的推导类型还是数组
- 那么表达式结果直接返回地址
- 只有真正取到标量元素时才 load

## 5.7 补齐局部数组初始化时缺失元素的零填充

局部数组初始化里，像：

```c
int a[4] = {1, 2};
```

后两个元素应该自动补零。

现在 `CodeGen` 会对**局部数组**做缺项零填充。

但对**全局数组**，修复后不再展开成一大串显式 store，而是依赖 `.bss` 的零初始化。这样可以避免超大全局数组把编译期内存直接打爆。

## 6. 除了 CodeGen，还修了哪些配套问题

虽然这个 README 重点是后端，但真正跑通测例不只靠 `CodeGen`。

还补过这些配套问题：

- `main.cpp`
  让 `--dump-asm` 模式在不需要解释执行时直接退出，避免“汇编生成成功却被解释器路径拖成 compile fail”。
- `AsmEmitter`
  修了函数标签唯一性、超大栈帧的 `sp` 调整、栈上传参与帧布局。
- `sylib.s`
  修了 `getarray/getfarray/putarray/putfarray` 里跨 `scanf/printf` 错用 caller-saved 寄存器的问题。
- `asm_test.sh`
  修了单测/目录模式、QEMU 栈大小、返回码附加格式。

## 7. 当前生成的 `.s` 最后是谁跑出来一个结果

这里可以把角色分清楚：

### 1. 你的编译器

负责把：

- `SysY`

翻译成：

- `RISC-V .s`

### 2. `riscv64-linux-gnu-gcc`

负责把：

- 你的 `case.s`
- 运行时库 `sylib.s`

变成：

- 一个 RISC-V Linux 可执行文件

它做的是汇编和链接，不负责语义翻译。

### 3. `qemu-riscv64-static`

负责真的执行这个 RISC-V 可执行文件。

也就是说，**最后那个“跑出来的结果”是 QEMU 执行链接后的 RISC-V ELF 得到的结果。**

## 8. 这个“结果”具体是什么

当前评测里，“结果”由两部分组成：

- 程序打印到标准输出的内容
- 程序的退出码

脚本会把它们拼接成：

```text
stdout 的全部内容
退出码
```

例如程序什么都不输出，最后 `return 3;`，那么实际结果文件大致就是：

```text
3
```

如果程序输出：

```text
114514
```

并且最后返回 `0`，结果文件会是：

```text
114514
0
```

## 9. 怎么判断它是否正确

判断标准非常直接：

- 看 `src/test_output/.../asm/out/<case>.out`
- 和 `test/.../<case>.out` 做文本比较

如果两者完全一致，就判这个用例通过。

脚本层面对应的是：

- `PASS`：输出和返回码都匹配
- `FAIL(compile)`：编译器没成功生成 `.s`
- `FAIL(link)`：`.s` 和 `sylib.s` 没成功链接
- `FAIL(diff)`：程序跑出来了，但结果和标准答案不同
- `TIMEOUT(...)`：超时

所以对于这个后端来说，“是否正确”最终不是看汇编长什么样，而是看：

1. 能不能生成合法 `.s`
2. 能不能被汇编并链接
3. QEMU 跑出来的标准输出和退出码是否与标准答案一致

## 10. 当前状态

当前这版直接后端已经能通过：

- `test/2026/functional`
- `test/2026/h_functional`

也就是说，它已经满足“直接翻译、能跑就行”的阶段目标。

后面如果要继续往上做，可以考虑：

- 加 Machine IR
- 做寄存器分配
- 减少无意义 load/store
- 做更好的全局初始化与常量池管理
- 做更完整的 ABI 细节收敛

但那已经是“优化和工程化”的下一阶段，不是这版 README 介绍的重点。
