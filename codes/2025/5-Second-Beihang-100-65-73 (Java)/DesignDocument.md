# Design Document of OHM Compiler

## 概述

本项目为 SysY2022 语言（C 语言子集）到 RISC-V 或 Arm v8 汇编代码的编译器，主体部分用 java 语言实现（见 `src` 目录），
并辅以 bash 和 python 脚本实现的自动化测试工具（见 `scripts` 目录）。

整个编译器系统采用前、中、后三端设计，其中
前端负责分析源文件的词法、语法、语义，以生成 LLVM IR；
中端基于 LLVM IR 的 SSA 结构进行多轮机器无关优化；
后端将 LLVM IR 翻译成汇编代码并进行寄存器分配以及机器相关优化。

## 运行控制

系统运行需要在命令行中指明输入输出文件路径，并选择优化等级、是否输出 IR 等，具体的参数设置规范参见项目根目录下的 `README.md`。

## 技术实现

### 前端

前端主要由词法分析器（`Lexer`）、语法分析器（`SyntaxParser`）、语义分析器（`Visitor`）三个单例执行类和相关数据结构类组成。

主要工作流程为：解析源代码形成 `Token` 流、分析 `Token` 流建立语法树、遍历语法树生成 IR 代码。

我们采用了较为标准的 LLVM IR，指令分类依照官方分类进行组织，并设置了抽象父类 `Value`，以维护使用（`Use`）关系、维护类型系统。

值得注意的是，我们依照 C 语言标准设计了较为完善的类型系统（详见 `src/frontend/ir/objecttype`），这为我们后续的中端优化和后端解析奠定了基础。

我们的所有分析器均为手写，并未采用自动化生成工具。

#### 词法分析
- 目的是将字符串形式的代码转化为存储代码信息的有意义的数据结构。
- 初步解析传入的源文件，将文本分解为词法单元（Token）列表形式。

#### 语法分析
- 根据 SysY 语法对词法单元列表进行解析，将其划分为语法单元，并通过递归下降的方法建立抽象语法树。
- 在这一步我们对标准 SysY 语法进行了一些改写，不影响原意的情况下消除了左递归、将一些类似的单元抽象出共同父类，以便后续解析。

#### 语义分析
- 遍历上一步建立的语法树，建立符号表、生成中间代码。
- 中间代码的框架结构大体为：IRModel -> Function -> BasicBlock -> Instruction，左侧为最高层级，层层包含。

### 中端

中端由多个优化 PASS 组成，每个常规优化 PASS 都是输入 LLVM IR，输出 LLVM IR，
除了一些 PASS 对前置分析 PASS 的依赖之外，可以任意进行排序而不影响正确性，
不过经过精心排序之后前置的 PASS 会给后续 PASS 创造更多优化机会，从而达到更好的效果。

中端的 PASS 在 `OptimizationHandler` 类中进行调用。

#### 常规优化

- Function inlining 函数内联
  - 将非递归函数内敛到调用者中，避免函数调用带来的跳转和栈操作，同时为后续优化创造机会
  - 做完函数内联删掉不再有用的函数定义，减轻后端翻译负担
- GlobalValueSimplify 全局对象简化
  - 全局对象简化，包括将没有被更新过的全局对象直接用初值替代，以及【局部化】
  - 局部化，即将所有仅在一个函数中被使用的全局对象变成对应函数中的局部对象，以便后续 mem2reg 操作
  - 全局对象简化之后也会删去没用的全局变量定义
- DeadBlockRemove 死块删除
  - 移除无用的基本块，简化代码
- CFG
  - 构建控制流图
  - 实现了对函数控制流图（CFG）进行支配树（Dominator Tree）和支配边界（Dominance Frontier）的计算
- Mem2Reg
  - 将内存操作转化为虚拟寄存器操作
  - 基于 alloca 指令进行，构造其 def-use 关系，简化 load 和 store 指令
  - 这部加入 phi 指令，实现非数组对象不基于内存操作的 SSA
- ArrayFParamMem2Reg
  - 针对数组形参的 mem2reg
  - 传入的数组形参实际上是数组首元素的指针，理论上不会被修改，因此 load 的结果就应该是第一次 store 的内容
- ConstFold 常数折叠
  - 对计算指令进行简化，可以在编译时进行的计算就在编译时直接算出
- FuncMemorize （递归）函数记忆化
  - 对于纯计算递归函数，通过添加全局数组和哈希函数减少递归运算
- LoopAnalysis 循环分析
  - 对中间代码的控制流进行分析，找出其中的循环结构以便后续优化
- LoopSimplify 循环简化
  - 对循环结构进行矫正，矫正为preHeader - Header - body - Latch的形式
- LCSSA 循环闭环静态单赋值化操作
  - 便于此后的循环优化
- TripCountAnalysis 循环迭代次数分析
  - 分析循环的迭代次数，用于后续的循环优化
- LoopUnroll 循环展开
  - 将循环结构变成顺序结构以创造更多优化机会
- LoopMemoryLift 循环内存操作外提
  - 将循环中与迭代变量无关的内存操作提出到循环外
- DeadCodeRemove 死代码删除
  - 结果没有被用到的代码应该被删掉
  - 例如：没有被用到的数组（DeadArrayRemove）、没有被用到的返回值（DeadRetRemove）、无影响的函数（DeadFunctionRemove）、没有被用到的参数（DeadArgumentRemove）等
- Global Value Numbering 全局值编号
  - 同样的计算只进行一次，避免重复计算
- Global Code Motion 全局值移动
  - 把指令尽可能放到较好的位置如循环外，同时能修正错误的指令顺序
- SimplifyBranch 分支简化
  - 将只有一种可能的条件判断块简化掉
- MergeBlock 基本块合并
  - 将多个跳转关系比较简单的基本块合成一个
- LoopSimplify 循环简化
  - 将循环累加变成乘法以减少循环操作
- RemoveUseLessLoop
  - 删除仅剩下迭代对象的无意义循环
- DetectTailRecursive
  - 识别尾递归
- StoreCSE
  - 消除跨基本块的冗余 store 指令
- LoadCSE
  - 消除跨基本块的冗余 load 指令
- StoreToLoadForwarding
  - 用 store 的 value 替换 load
- PureFunctionAnalysis 纯函数分析
  - 分析函数的各种属性，例如写全局、读全局等
- AliasAnalysis 别名分析
  - 类型敏感的指针别名分析
- SCEVAnalyse 标量演化
  - 分析标量的演化过程，为循环变量分析打基础
- SCEVSimplify 演化简化
  - 利用标量演化过程实现化简，如累加变乘

#### 中后端衔接

- RemovePhi
  - 将 phi 指令替换成 move 操作以方便后端翻译成汇编

#### 并行化优化

对于多次迭代之间没有依赖的循环，可以将循环体部分提出到一个单独的函数，
然后由预先写好的并行库函数（参考自往届优秀队伍 CMMC）进行调用。

#### 向量化优化

向量化是使用 SIMD（单指令多数据）指令来快速进行批量操作。

进行向量化首先要将循环展开成 4 轮迭代一组的大循环和一个处理剩余部分的尾循环，然后检测循环中的数据操作，
对于内存存取、乘加、乘减等操作，可以用 SIMD 指令进行批量操作以适配硬件优化。

中端主要负责检测向量化机会并生成伪指令，真正的 SIMD 指令要在后端依照架构特性进行生成。

### 后端

#### IrParser

IrParser的主要任务是将IR代码翻译为RISC-V代码。对于大多指令，在不考虑优化的情况下可对应进行翻译。较为特殊的有以下几个要点：
- 栈空间的分配。我们的设计是每个函数维护自己的栈，从栈顶到栈底依次存储函数内Call指令8个以后的参数（默认使用a0-a7进行传参），执行函数过程中使用alloca指令申请的栈空间，寄存器溢出占用的栈空间，ra。由于寄存器溢出占用的栈空间在寄存器分配之前无法计算，因此在IrParser中会计算剩下三个部分的栈空间大小，最后在寄存器分配部分完善栈空间的申请。
- 指令翻译。通过HashMap建立IR中虚拟寄存器与RISC-V中虚拟寄存器的映射关系。在翻译过程中尽量采用基本指令，从而避免拓展指令对寄存器值进行意外的修改，后续优化也会更加灵活。
- 指令降级。对乘、除、模等指令，转化为CPU周期更小的移位、加减等指令。

由于我们的后端适配了ARM以及RISC-V两种架构，因此在IrParser中会根据不同的架构进行不同的指令翻译。
为避免重复使用代码，我们将IrParser作为抽象类，其中实现包括栈分配等通用功能，而RISC-V和ARM的指令翻译则在各自的子类（IRParser4RV, IRParser4ARM）中实现。

#### RegAlloc

核心代码在 `src/backend/asm/regalloc/UnitedRegAllocator.java` 文件中，部分数据流分析方法分散在 `ASMBasicBlock` 等类中。

- 参考虎书的经典启发式图着色算法进行寄存器分配
- 总体流程是：进行初始化、分析调用关系、以函数为单位运行图着色算法、进行寄存器的调用保护、最终修正栈大小使得其 16 字节对齐
- 图着色部分包括活跃变量分析、建立冲突图、冲突图着色（简化、合并、冻结、溢出、重新开始）
- 函数调用保护部分，程序内的函数由于可以确认被调用者真正使用了哪些寄存器，可以直接在调用处保存需要保存的寄存器，而不是保存所有跨调用活跃的调用者保存寄存器；调用库函数则需要保存所有当前用到的调用者保存寄存器
- 一开始我们尝试用寄存器池分配局部寄存器、图着色分配全局寄存器，后来发现并无必要（效果并没有提升），因此直接统一成图着色。

#### Optimizer

- 主要负责对翻译后的汇编代码进行机器相关优化。我们认为中端优化相对来说是更为有效的，因此后端只进行了简单的指令选择与窥孔。
- 具体优化包括但不限于简化条件跳转、消除冗余load/store、合并乘法加法、指令向量化等。
